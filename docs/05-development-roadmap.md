# 研发落地任务拆解

> 本文档定义币安量化交易系统的开发路线图、任务拆解、团队配置、开发规范和成本估算。
> 总工期预估 16-24 周（4-6 个月），具体节奏视团队到位情况和技术 Spike 结果调整。

---

## 1. 项目里程碑规划

### 总览

```
Phase 1: 基础设施搭建      Week 1-4     ████░░░░░░░░░░░░░░░░░░░░
Phase 2: 核心交易引擎      Week 3-8     ░░████████░░░░░░░░░░░░░░
Phase 3: 策略开发与回测    Week 7-14    ░░░░░░████████████░░░░░░
Phase 4: 风控与监控系统    Week 11-16   ░░░░░░░░░░████████░░░░░░
Phase 5: 上线与优化        Week 15-20   ░░░░░░░░░░░░░░████████░░

里程碑标记:
  M1 (Week 4)  - 基础设施就绪，可连接交易所获取数据
  M2 (Week 8)  - 交易引擎可在 Testnet 下单
  M3 (Week 14) - 至少 2 个策略完成回测验证
  M4 (Week 16) - 风控系统上线，具备生产运行条件
  M5 (Week 20) - 小资金实盘稳定运行 4 周
```

### Phase 间依赖关系

```
Phase 1 (基础设施)
  │
  ├──→ Phase 2 (交易引擎) ──→ Phase 3 (策略开发)
  │                               │
  │                               ├──→ Phase 4 (风控监控)
  │                               │         │
  └───────────────────────────────────────→ Phase 5 (上线优化)
```

注：Phase 2 后半段与 Phase 3 前半段有 2 周重叠（策略开发可在引擎核心完成后并行启动），Phase 3 后半段与 Phase 4 有 4 周重叠。

## 2. 团队配置建议

### 2.1 所需角色

| 角色 | 人数 | 核心职责 | 参与 Phase |
|------|------|---------|-----------|
| 后端工程师 (Senior) | 1 | 交易引擎、数据管道、系统架构 | 1-5 |
| 后端工程师 (Mid) | 1 | API 集成、数据存储、监控系统 | 1-4 |
| 策略研发工程师 | 1 | 策略设计、回测框架、信号生成 | 2-5 |
| DevOps 工程师 | 0.5 (兼职) | 基础设施、CI/CD、部署运维 | 1, 4-5 |
| 风控/量化分析师 | 0.5 (兼职) | 风控模型、绩效归因、合规审计 | 3-5 |

### 2.2 人员能力要求

**后端工程师 (Senior)**
- 3+ 年 Python 后端开发经验
- 精通异步编程（asyncio）、WebSocket 长连接
- 有交易系统或金融系统开发经验优先
- 理解低延迟系统设计（内存管理、GC 优化）
- 熟悉 PostgreSQL / ClickHouse 等分析型数据库

**策略研发工程师**
- 量化金融/统计学/数学背景
- 精通 Python 数据栈（pandas、numpy、scipy）
- 有加密货币或传统金融量化策略开发经验
- 理解回测偏差（look-ahead bias、survivorship bias）
- 具备独立从论文/想法到可执行策略的全流程能力

**DevOps 工程师**
- 熟悉 Docker、Kubernetes 或 Docker Compose
- 有 Prometheus + Grafana 监控体系经验
- 了解网络安全基础（防火墙、VPN、密钥管理）
- 能配置 CI/CD 流水线（GitHub Actions / GitLab CI）

### 2.3 团队规模建议

| 模式 | 人数 | 周期 | 适用场景 |
|------|------|------|---------|
| 最小团队 | 2 人 | 6 个月 | 1 后端 + 1 策略，后端兼 DevOps，适合个人/小团队 |
| 推荐团队 | 3 人 | 4-5 个月 | 2 后端 + 1 策略，DevOps/风控兼职 |
| 加速团队 | 5 人 | 3-4 个月 | 全角色到位，并行度最高 |

## 4. 技术 Spike 清单

在正式开发前，需完成以下技术验证，以降低后续开发风险。每个 Spike 输出一份简短的技术报告。

### Spike 1: 币安 WebSocket 稳定性测试

| 属性 | 详情 |
|------|------|
| 预估工时 | 2 人天 |
| 优先级 | P0（阻塞 Phase 1） |
| 验证目标 | WebSocket 长连接稳定性、重连机制、消息延迟分布 |
| 方法 | 搭建测试脚本，同时订阅 20+ 交易对，连续运行 72h，记录断连次数、重连耗时、消息延迟 |
| 成功标准 | 72h 内断连 < 5 次；自动重连成功率 100%；消息延迟 p99 < 500ms |
| 风险 | 如不稳定，需评估备用数据源（如 CCXT、第三方聚合商） |

### Spike 2: 低延迟下单性能测试

| 属性 | 详情 |
|------|------|
| 预估工时 | 2 人天 |
| 优先级 | P0（阻塞 Phase 2） |
| 验证目标 | REST API 下单延迟、从信号到成交的端到端延迟 |
| 方法 | Testnet 环境，测量 1000 次下单的延迟分布（区分网络延迟 vs 交易所处理时间） |
| 成功标准 | REST 下单 p50 < 100ms，p99 < 300ms；服务器位于新加坡/东京（靠近币安服务器） |
| 风险 | 如延迟过高，需考虑服务器选址或使用 VPS 靠近交易所 |

### Spike 3: Python 异步并发性能

| 属性 | 详情 |
|------|------|
| 预估工时 | 1 人天 |

| 方法 | 模拟 50 个策略并发运行 + 100 个 WebSocket 连接，测量事件处理延迟和 CPU 使用率 |
| 成功标准 | 事件处理延迟 p99 < 10ms；CPU 单核占用 < 60% |
| 风险 | 如性能不足，关键路径考虑 Cython / Rust 绑定 |
